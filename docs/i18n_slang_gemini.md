# Full UI internationalisation via `slang` + Gemini

## Goal

Tap the language flag on **any** screen → every visible word switches to the
chosen language. No English fallbacks anywhere when a non-EN locale is active.
Targets the same 10 languages already supported by the backend recipe pipeline:

```
ru, en, es, fr, de, it, tr, ar, fa, ku
```

This document covers **static UI strings only** (app bar titles, tab labels,
section headers, hints, tooltips, loading screens, snackbars, plurals,
accessibility labels). Dynamic recipe content (name, ingredients, instructions)
is already translated server-side by the mahallem `translate-recipe` pipeline
and is out of scope here.

## Decision

| Concern | Pick | Why |
|---|---|---|
| String catalog framework | [`slang`](https://pub.dev/packages/slang) | Codegen → typed `t.foo.bar` getters, full CLDR plurals, runtime locale switch via `LocaleSettings.setLocaleRaw`, build fails if any locale is missing a key. |
| Bulk translator (build-time) | Gemini 2.5 Flash via REST | Free tier (1M tokens/day) covers our ~30-string × 9-locale job many times over. Key already provisioned in `mahallem_ist/.env`. |
| Locale switching | Existing `appLang` `ValueNotifier` | Listener calls `LocaleSettings.setLocaleRaw(appLang.value.name)`. |
| Material widget strings | `flutter_localizations` SDK | Date pickers / default tooltips localised by Flutter itself for all 10 codes. |
| RTL flip (ar / fa / ku) | `Directionality` driven by `appLang` | One wrapper around `MaterialApp.builder`. |

We deliberately do **not** use `slang_gpt` (the Pub package). It depends on
OpenAI by default and adds a dev dependency we don't otherwise need. Instead
we ship a tiny Dart script (`tool/translate_strings.dart`) that reads
`.env_gemini`, calls the Gemini REST endpoint, and writes the per-locale slang
JSON files. Same outcome, fewer dependencies, full control over the prompt.

│   │   ├── strings_es.i18n.json # generated by tool/translate_strings.dart
│   │   ├── strings_fr.i18n.json
│   │   ├── strings_de.i18n.json
│   │   ├── strings_it.i18n.json
│   │   ├── strings_tr.i18n.json
│   │   ├── strings_ar.i18n.json
│   │   ├── strings_fa.i18n.json
│   │   └── strings_ku.i18n.json

`appLang.name` (e.g. `'es'`) maps 1:1 to slang's locale code and to the JSON
filename suffix.

## How locale switching works at runtime

1. User taps `LangIconButton` → `cycleAppLang()` → `appLang.value = next`.
2. A single listener installed in `main.dart` calls
   `LocaleSettings.setLocaleRaw(appLang.value.name)`.
3. slang updates `LocaleSettings.currentLocale`, the top-level
   `TranslationProvider` (an `InheritedWidget`) rebuilds its descendants.
4. Every call site uses `final t = Translations.of(context);` and reads
   `t.tabRecipes`, `t.recipeTitle`, etc. — all rebuild automatically.
5. For RTL, `MaterialApp.builder` wraps the child in `Directionality(
   textDirection: appLang.value == AppLang.ar /* fa, ku */ ? rtl : ltr)`.

## How Gemini fills the 8 non-EN/RU locales

`tool/translate_strings.dart`:

1. Reads `recipe_list/.env_gemini` for `GEMINI_API_KEY`.
2. Loads `lib/i18n/strings.i18n.json` (English).
3. For each target lang in `[es, fr, de, it, tr, ar, fa, ku]`:
   - POSTs to
     `https://generativelanguage.googleapis.com/v1beta/models/gemini-2.5-flash:generateContent?key=…`
   - Prompt: "Translate the following JSON values to {language name}. Keep keys
     and ICU plural placeholders untouched. Output **only** valid JSON, same
     shape as input."
   - Validates JSON shape: every key in input must appear in output, no extras,
     plural blocks intact.
4. Writes `lib/i18n/strings_<code>.i18n.json`.

Run it once after editing the base JSON:

```sh
cd recipe_list
dart run tool/translate_strings.dart
dart run slang
```

`flutter test` then validates the result via tests below.

## Key lifecycle / security

- `GEMINI_API_KEY` lives only in `recipe_list/.env_gemini` (chmod 600).
- `.env_gemini` is gitignored at repo root — `git check-ignore` passes.
- The key is **never** bundled into the iOS / Android binary. It's used only at
  developer build time by `tool/translate_strings.dart`.
- The runtime app makes zero requests to Gemini.

See `docs/todo/i18n_slang_gemini.md` for the per-chunk test plan. Final assertions:

- `test/i18n_completeness_test.dart`: for every `AppLang` value, the slang
  `Translations` instance produces a non-empty string for every documented key,
  and no value equals its English counterpart unless explicitly tagged
  (e.g., `youtube`, brand names).
- All existing widget tests continue to pass (`recipe_card_test`,
  `recipe_list_page_test`, `recipe_api_test`).

## Out of scope (already working / handled elsewhere)

- Recipe content translation: server pipeline at
  `local_user_portal/utils/translate-recipe.js`. Already covers all 10 langs.
- Per-`(id, lang)` cache: existing sqflite + Postgres layers.
- Lang button visibility / shared AppBar: shipped via `AppPageBar`.
- Re-fetch on lang change without reshuffle: shipped in
  `RecipeListLoader._onLangChanged`.
